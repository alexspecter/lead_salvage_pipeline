PROJECT OBJECTIVE
Build a local Python pipeline that:
	•	Accepts a messy CSV file containing lead data
	•	Cleans and normalizes data deterministically
	•	Routes unresolved rows to a local LLM (Llama 3.1 via MLX)
	•	Outputs:
	◦	final_output.csv
	◦	reject_store.csv
	◦	log.csv
All rows must be accounted for exactly once.

OPERATING CONSTRAINTS
The system must satisfy the following constraints:
	•	Runs entirely locally (MacBook M4 Max, 64GB Unified Memory)
	•	**Hardware Loop:** The Agent writes code; the Human runs it. The Agent must request logs to verify success.
	•	**Memory Guard:** Phase 2 must strictly monitor RAM. If usage > 95%, the process must PAUSE or ABORT. Do not rely on macOS swapping.
	•	No network calls (Air-gapped safety).
	•	LLM inference via mlx-lm library.
	•	Deterministic behavior where possible.
	•	Full audit trail per run.
	•	Safe for PII (no leakage in logs).
Violating any of these is a failure.

HIGH-LEVEL PIPELINE FLOW
SETUP (Generate Test Data) ↓ INPUT VALIDATION ↓ PHASE 1 (Deterministic) ↓ **STOP: REQUIRE 100% UNIT TEST PASS** ↓ PHASE 2 (Semantic LLM) ↓ PHASE 3 (Merge/Verify)
Each phase must be isolated and callable independently.

DATA CONTRACTS (MANDATORY)
ROW SCHEMA (USED IN ALL PHASES)
Each row must conform to this structure:
row_id: UUID (permanent) run_id: UUID (job-level) raw_data: dictionary (original values) clean_data: dictionary (cleaned values) status: ENUM { CLEAN, AI_REQUIRED, REJECTED } failure_reason: optional ENUM confidence_score: float between 0.0 and 1.0 is_duplicate: boolean duplicate_of: UUID or null

LOG EVENT SCHEMA
Every action must emit a structured log event:
run_id: UUID row_id: UUID or null phase: string action: string before: string or null after: string or null reason: string or null confidence: float or null timestamp: ISO-8601
No free-form logging is allowed.

PHASE 0 — SETUP & VALIDATION
FILES: core/validator.py, phase0_setup/generator.py
RESPONSIBILITIES:
	•	**Garbage Generator:** Create `generate_messy_leads.py` to produce deterministic "bad" data (emojis, dupes, bad dates) for testing.
	•	Generate run_id and create run directory.
	•	Copy original CSV.
	•	Validate CSV readability.
	•	Normalize headers (lowercase, strip spaces).
	•	Validate required columns (At least one of: email, phone).
If validation fails, abort immediately with a logged error.

PHASE 1 — DETERMINISTIC CLEANING
ENTRY POINT: phase1_deterministic/runner.py
EXECUTION ORDER (MUST FOLLOW EXACTLY):
	1	Load CSV into pandas DataFrame
	2	Assign row_id UUID to each row
	3	Normalize fields using normalizers: date, phone, email, name, job title
	4	Generate row fingerprint (email|phone or row_id)
	5	Detect duplicates (Group by fingerprint, mark subsequent as REJECTED)
	6	Assign deterministic confidence score
	7	Route rows (CLEAN, AI_REQUIRED, REJECTED)
	8	Log every transformation

**CRITICAL GATE:**
Before proceeding to Phase 2, the Orchestrator must run `tests/test_phase1.py`.
If ANY unit test fails (regex logic, duplication logic), the pipeline must HALT.
Do not waste GPU cycles on broken logic.

NORMALIZER RULES:
Each normalizer must process one field and return a tuple: (normalized_value, field_status, reason)
Never invent data. Never guess.

PHASE 2 — SEMANTIC CLEANING (LLM)
ENTRY POINT: phase2_semantic/runner.py
MODEL CONSTRAINTS:
	•	Model: Llama 3.1 70B 4-bit (approx 40GB VRAM)
	•	Backend: mlx-lm
	•	Temperature: 0
	•	No streaming
	•	**MEMORY GUARD:**
		◦	Implement `phase2_semantic/memory_guard.py` using `psutil`.
		◦	Before loading model: Check available RAM.
		◦	During batch processing: Check RAM every chunk.
		◦	**Rule:** If RAM usage > 95% (approx 60GB), STOP loading/processing.
		◦	Log "CRITICAL_MEMORY_WARNING" and dump current progress to disk.
		◦	Do not allow the OS to initiate heavy swap (thrashing).
	•	Chunking: Start with `chunk_size=5`. Dynamically reduce if memory pressure rises.

PROMPT RULES:
	•	Do not invent missing data.
	•	Output valid JSON only (no markdown, no backticks, no preamble).
	•	If `mlx-lm` fails or crashes, catch the exception, log it, and mark rows as REJECTED (Reason: MODEL_CRASH).

EXECUTION STEPS:
	1	Load model once (Guarded by Memory Check)
	2	Load AI_CLEAN rows (status=AI_REQUIRED)
	3	Chunk rows for batch processing
	4	For each chunk:
	◦	Check Memory Guard
	◦	Format prompt -> Run inference -> Parse JSON
	◦	Validate output schema (Check for hallucinations)
	◦	Update clean_data and status
	5	Log all changes
	6	Shut down model cleanly and FORCE garbage collection

PHASE 3 — MERGE & VERIFY
ENTRY POINT: phase3_merge/runner.py
MERGE RULES:
	•	Merge P1_CLEAN and AI_CLEAN (all rows with status=CLEAN) → final_output.csv
	•	All rows with status=REJECTED → reject_store.csv
	•	Use row_id as the merge key

VERIFICATION RULES:
	•	Count original input rows
	•	Count final_output rows + reject_store rows
	•	Verify: input_count == output_count + reject_count
	•	Verify: no row_id appears in both final_output and reject_store
	•	Verify: every original row_id appears exactly once across outputs

If verification fails: DO NOT write outputs. Log critical discrepancy. Raise VerificationError.
